## 서문: 루미나의 나눔과 「상선약수(上善若水)」

**『나눔을 통한 성장』**은 루미나플랫폼이 **독자적으로 선언하는 철학**이며, 단순한 윤리 구호가 아니라 **아키텍처의 제1원리**다. **「상선약수」** 에 대한 해석과 실천은 **멘토의 고유한 경영 철학**으로 정립하고, 과거 사업·동업 사례와 **섞지 않는다.** 노자가 말한 물의 덕—가장 낮은 곳으로 흘러 만물을 이롭게 함—은 기술 언어로 번역될 때 다음과 같이 귀결된다.

- **낮은 진입장벽:** 소상공인·인디 브랜드가 **무료 또는 저비용**으로 AI 에이전트 능력에 닿을 수 있는 **게이트웨이·티어·크레딧** 구조.
- **투명한 흐름:** 수익과 수수료가 **어디로, 어떤 목적으로** 흐르는지 파트너가 **검증 가능한 단위**로 확인.
- **공생 우선:** 상담·정산·추천을 수행하는 AI는 **플랫폼 이익 단일 극대화**가 아니라 **상생(Win-Win)** 을 최우선 가치로 두는 **행동 강령**에 묶인다.

18년의 정신 **온(穩)·준(準)·쾌(快)** — 견고함·정확함·신속함 — 은 본 전략에서 **신뢰(온)·정산·수치의 정확성(준)·응답과 처리 속도(쾌)** 로 구현된다.

## 1. [나눔의 기술 구조] 상생형 API 설계안

### 1.1 목표

소상공인 파트너가 **자체 쇼핑몰·POS·앱**에서 루미나의 AI 에이전트(브랜딩 카피, 제품 분석, 렌더 큐 등)를 **무료(Community)·저비용(Growth)·계약 기반(Scale)** 으로 연동할 수 있게 한다. “비싼 AI”를 **나눔형 인터페이스**로 낮추는 것이 제2호 브랜딩의 기술적 핵심이다.

### 1.2 논리 계층

| 계층 | 역할 |
|------|------|
| **파트너 애플리케이션** | 주문·재고·고객 접점; 루미나 호출은 서버 사이드 권장. |
| **API Gateway** | 라우팅, TLS 종단, WAF, 지역 라우팅. |
| **인증·권한** | `Authorization: Bearer <partner_api_key>` + 스코프(RBAC). |
| **티어·Rate limit** | `community` / `growth` / `scale` — 분당 호출·월 포함 단위로 공정 사용. |
| **에이전트 런타임** | 동기 `invoke` vs 비동기 `jobs` 분기. |
| **Usage·Credits** | 호출 단위·크레딧 차감; 파트너 대시보드와 동일 소스. |

코드 앵커: `lib/partners/sharing-integration-api.ts` (`LUMINA_SHARING_API_VERSION`, `LUMINA_SHARING_TIER_DRAFT`, `LUMINA_SHARING_API_ROUTES_DRAFT`).

### 1.3 엔드포인트 초안 (REST)

| 구분 | 메서드·경로 | 설명 |
|------|-------------|------|
| 가용성 | `GET /v1/health` | 게이트웨이·하위 의존성 헬스. |
| 능력 공개 | `GET /v1/capabilities` | 해당 키에 허용된 `agentId`, 버전, 제한 사항. |
| 실행 | `POST /v1/agents/{agentId}/invoke` | 단발 추론·생성; 장시간 작업은 `202` + `jobId` 반환 정책 가능. |
| 비동기 | `GET /v1/jobs/{jobId}` | 렌더·대용량 작업 상태. |
| 투명성 | `GET /v1/usage/summary` | 기간별 호출·크레딧 잔여 — **나눔의 가시화**. |

### 1.4 무료·저비용의 기술적 정의

- **무료(Community):** 월 일정 **포함 호출·크레딧** 내에서 과금 0; 초과 시 **성장 티어 안내** 또는 소액 과금(정책 확정).  
- **저비용(Growth):** 월 구독 또는 사용량 기반 **상한 명시**, 숨은 요금 금지 UI/API.  
- **Scale:** SLA, 전담 온보딩, 별도 계약 — **동일 API 표면**으로 일관성 유지.

### 1.5 비기능 요구사항

- **멱등성:** `Idempotency-Key` + 서버측 중복 제거로 **이중 청구·이중 작업** 방지.  
- **웹훅:** `callbackUrl` + HMAC 서명으로 결과 전달 시 위변조 방지.  
- **감사 로그:** 파트너 ID(해시), agentId, jobId, 타임스탬프, 결과 코드 — **민감 본문은 최소 저장**.

### 1.6 내부 매핑

공개 `/v1` 은 기존 Next.js `app/api/*` 기능에 대한 **파사드**로 두어, 한 곳의 비즈니스 로직을 **웹 UI와 파트너 API가 공유**하게 한다.

## 2. [투명 정산 시스템] 실시간 대시보드 기획

### 2.1 “실시간”의 정의

- **기술적 의미:** Ledger 또는 집계 파이프라인이 갱신되는 시점에 맞춰, 파트너가 **지연 없이(목표: 수 초~수 분 내)** 스냅샷을 조회할 수 있음.  
- **법·운영적 의미:** 최종 확정 정산은 **약관상 확정일·감사 절차**에 따름; UI에는 **“확정 전 추정” vs “확정”** 상태를 구분 표시한다.

### 2.2 데이터 모델 (개념)

1. **매출·귀속:** 파트너 신고 또는 연동 집계 → 기간별 `grossRevenueWon`, `attributedRevenueWon`.  
2. **성공보수:** `computeRevenueShareSettlement` (`lib/partners/revenue-share.ts`) — 요율·귀속 기반.  
3. **투명 스냅샷:** `buildSettlementTransparencySnapshot` (`lib/partners/settlement-transparency.ts`) — 항목별 라인 + 서술.  
4. **영속 Ledger:** `ledgerId` 및 행 단위 이벤트(향후 DB); 대시보드는 **읽기 전용 API**.

### 2.3 대시보드 화면 구성

| 영역 | 내용 |
|------|------|
| 기간 선택 | 주·월·분기; 비교 기간 선택 시 **전기 대비** 참고 지표. |
| 요약 | 귀속 매출, 성공보수 합계, 적용 요율, (선택) 기준 대비 uplift. |
| 배분明細 | 각 `SettlementTransparencyLine` — 금액·`explanationKo` 동시 표시. |
| 나눔 서술 | 수수료가 **플랫폼 운영·파트너 생태계 재투자** 등으로 쓰이는 이유(정책 문구, 법무 확정). |
| 메타 | `generatedAt`, 정책 버전 ID, `ledgerId` 링크(권한 있는 사용자만). |

### 2.4 API 연동 (현 구현)

- `GET /api/partners/settlement-transparency?partnerId=&periodStart=&periodEnd=&grossRevenueWon=`  
- 선택 쿼리: `attributedRevenueWon`, `sharePercent`  
- 응답: `snapshot` JSON — 프론트는 테이블·카드·타임라인 뷰로 소비.

### 2.5 환경 변수 (배분 비율 초안)

| 변수 | 설명 |
|------|------|
| `LUMINA_TRANSPARENCY_OPS_PERCENT_OF_FEE` | 성공보수 중 플랫폼 운영 배분(초안 기본 60). |
| `LUMINA_TRANSPARENCY_REINVEST_PERCENT_OF_FEE` | 파트너 나눔·접근성 풀(초안 기본 40). |

합계 100% 유지; 변경 시 **정책 버전**을 스냅샷 메타에 기록한다.

### 2.6 보안

세션·API 키·IP 제한; **다른 파트너 데이터 절대 혼선 없음**; 감사 추적 가능.

## 3. [AI 에이전트 행동 강령] 공생 최우선 로직 가이드라인

상담·추천·정산 안내를 수행하는 LLM·룰 기반 에이전트는 **플랫폼 매출 단기 극대화**보다 **공생·투명성·안전**을 상위에 둔다.

### 3.1 최우선 가치 (우선순위)

1. **진실성·준법:** 허위 효능·근거 없는 수치·의료 주장 금지.  
2. **파트너 이익:** 동일 정보로 **파트너 매장·브랜드 성장**에 유리한 선택을 먼저 제안.  
3. **투명성:** 수수료·한도·정산 구조를 **숨기지 않고** 이해 가능한 언어로 설명.  
4. **플랫폼 지속가능성:** 장기적으로 파트너 생태계를 망가뜨리는 단기적 착취 제안 금지.

### 3.2 판단 보조 로직 (구현 지침)

| 단계 | 내용 |
|------|------|
| **입력 분류** | 질의가 정산·요금·한도·기술 연동·브랜딩 중 무엇인지 태깅. |
| **정책 조회** | 해당 티어의 공개 한도·요율·FAQ 스니펫을 RAG 또는 정적 컨텍스트로 주입. |
| **충돌 시** | 파트너에게 불리한 암시적 업셀보다 **명시적 비교표** 제공. |
| **출력 검증** | 금지 패턴(과장, 허위 할인, 미확정 수치를 확정처럼) 필터 — 서버측 후처리 권장. |
| **로그** | 모델 버전, 정책 버전, 세션 ID(비식별); 개인정보 최소 저장. |

### 3.3 시스템 프롬프트에 포함할 핵심 문장 (예시)

- “당신은 루미나의 상담 에이전트이다. 답변은 **파트너 상생**과 **정직한 정보**를 최우선으로 한다.”  
- “수수료·정산은 **추정일 수 있으며** 확정은 약관과 시스템 알림을 따른다고 안내한다.”  
- “불확실하면 추측하지 말고 **확인 경로(대시보드·고객 지원)** 를 안내한다.”

### 3.4 금지·주의

- 경쟁사 비방, 파트너 내부 정보 무단 추론.  
- 정산 금액을 **확정**으로 단정(상태 구분 없이).  
- Human-Safety·Diamond 본질 등 **기존 루미나 브랜드 규범** 위반 카피.

### 3.5 정산 전용 에이전트

- 입력: 기간, 매출 숫자, 공개 요율.  
- 출력: **라인 아템 설명** + `settlement-transparency` 스키마와 호환되는 요약만; **법적 조언 아님** 문구 필수.

## 4. [직원 훈시문] 비전 선포 — 대표이사의 나눔 철학

동지 여러분,

루미나는 18년의 현장과 법·세무의 정밀함 위에 서 있고, 이제 그 길 위에 **AI**를 얹었다. 우리가 만드는 것은 단순한 소프트웨어가 아니다. **루미나가 세상에 처음 제시하는** **나눔을 통한 성장**이라는 **철학**이, **기술·정산·화면**의 언어로 쓰이는 자리다.

**물은 낮은 곳으로 흐른다.** 우리의 기술도 먼저 **작은 가게·용기 있는 브랜드**의 발밑에 이르게 하라. API 한 줄, 버튼 한 개, 정산 표의 숫자 하나가 **누군가의 문을 닫는 낙인**이 되지 않게 하라. **투명함**은 미덕이 아니라 **신뢰의 공학**이다. **정확함(準)** 은 호의가 아니라 **존중의 형식**이다.

우리의 AI는 **사장님들을 더 잘 속이기 위한 도구**가 아니라, **함께 크는 상생**을 돕는 동료가 되어야 한다. 코드를 짜고, 카피를 고르고, 장표를 설계하는 모든 순간에 스스로에게 물으라. **“이것이 파트너에게 이로운 나눔인가?”** 그 질문에 부끄럽지 않은 하루가 곧 루미나의 **품격**이다.

**온(穩)** 으로 흔들리지 말고, **준(準)** 으로 흐리지 말고, **쾌(快)** 하게 짐을 덜어 주어라. 이 길은 **모두가 사장**이라는 약속의 길이며, 여러분의 손끝에서 **대중적 럭셔리**가 현실이 된다.
